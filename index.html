Artifacts: Raspberry Pi Thermostat Project

The thermostat project began as a simple hardware prototype and gradually evolved into a complete embedded computing system. Each version expanded the design both conceptually and technically, adding new logic, structure, and capabilities. This section shows the development from a foundational Python script into a fully engineered application that incorporates algorithms, modular design, and database integration.

  • Original Thermostat Project (CS 350)
    The first version was created for CS 350: Emerging Systems Architectures and Technologies. It established the basic functionality of the thermostat: reading temperature data from an AHT20 sensor, displaying the output on a 16x2 LCD, and controlling red and blue LEDs to represent heating and cooling. The code directly handled all sensor, LED, and display operations in one file without modular structure. While functional, it lacked separation of concerns and relied heavily on sequential logic, which made future expansion difficult.
  • Version 2 – Software Design & Engineering
    Version 2 focused on improving software organization and maintainability. The code was restructured into distinct classes, including a ManagedDisplay class for LCD control and a TemperatureMachine class that implemented a clear state machine. Threads were introduced to handle asynchronous display updates, and locks were used to prevent resource conflicts on the I²C bus. This milestone aligned with the software design and engineering outcome by turning a procedural prototype into a modular, object-oriented program that follows clear state-driven design principles.
  • Version 3 – Algorithms & Data Structures
    The third version added algorithmic reasoning and efficiency improvements. Sensor readings were processed through a moving average algorithm to smooth noise and improve stability. A hysteresis system was added to reduce LED flicker by creating a temperature buffer between on and off states. The program logic was updated to handle time-based events and manage scheduled temperature adjustments. These changes introduced computational efficiency, better real-world responsiveness, and more intelligent control logic, demonstrating an understanding of how algorithmic design improves performance and user experience.
  • Version 4 – Database Integration
    The final version integrated a local SQLite database to enable data persistence. Each temperature reading and thermostat event was logged automatically with timestamps, allowing for long-term analysis of temperature trends and user adjustments. The database logic was implemented in a separate ThermostatDatabase class that manages connections, writes, and safe commits. This addition showed how embedded systems can leverage databases for efficient data management, bridging hardware control with long-term digital storage and analysis.

Narratives and Reflections

Each narrative expands on the learning process behind the code revisions and explains how the enhancements align with core computer science principles. Together, they document the full development cycle—from early design thinking to algorithmic refinement and data management.

  • Milestone Two – Software Design and Engineering Narrative
    This narrative explains how I refactored the original CS 350 thermostat code to follow modular design practices. It discusses the process of encapsulating functionality into reusable classes, applying threading to manage concurrent processes, and improving the code’s readability and scalability. It also reflects on how these changes supported secure and stable system operation by reducing direct hardware conflicts and enforcing clearer logic boundaries.
  • Milestone Three – Algorithms and Data Structures Narrative
    This narrative focuses on how algorithmic thinking transformed the thermostat’s behavior. It explains the reasoning behind adding moving averages to stabilize readings, hysteresis logic to prevent frequent LED toggling, and scheduling algorithms for adaptive control. It also details how these updates improved performance and reliability, linking them to theoretical foundations in data processing and algorithmic optimization learned earlier in the program.
  • Milestone Four – Database Integration Narrative
    This final reflection explores the integration of persistent data storage using SQLite. It discusses designing the schema for temperature and event logging, implementing efficient database writes, and handling potential concurrency issues. The narrative connects these technical updates to broader database design principles such as normalization, indexing, and transaction safety, emphasizing how structured data management completes the system’s full lifecycle of sensing, processing, and storage.

Security Considerations

While the thermostat operates locally, design choices such as input validation, I²C lock control, and exception handling were added to improve reliability and prevent system failure. These features reflect a security mindset that anticipates and mitigates risks such as sensor errors, data corruption, and device miscommunication, ensuring system integrity even in unexpected conditions.
